## 一、核心假设 (The One Thing to Prove)

**我们要验证的核心假设：**

> "AI 能够将散乱的用户反馈自动聚类并生成准确的需求主题，其效率和质量超过人工整理 10 倍以上。"

**如果这个假设不成立，产品的所有其他功能都没有意义。**

---

## 二、目标用户与使用场景

### 目标用户
- 中小企业的产品经理 (PM)
- 客户成功 (CS) 团队负责人
- 创业公司创始人

### 典型使用场景
**现状痛点：**
- PM 每周要从微信群、Excel、邮件中整理 50-200 条反馈
- 花费 4-6 小时手动分类、去重、提取关键需求
- 无法量化"哪些需求最该优先做"

**期望解决：**
- 10 分钟导入 100 条反馈
- 系统自动聚类成 10-15 个需求主题
- 一键查看按优先级排序的需求列表

## 三、核心数据结构 (Data Model First)

在写代码之前，必须先定义清晰的实体关系：

```
┌─────────────┐
│   客户      │ (Customer)
│  - ID       │
│  - 名称     │
│  - 付费等级 │ [普通/付费/大客户/战略]
└──────┬──────┘
       │ 1:N
       ↓
┌─────────────┐
│ 原始反馈    │ (Feedback)
│  - ID       │
│  - 内容     │
│  - 来源     │ [手动录入/Excel导入]
│  - 提交时间 │
│  - 客户ID   │ (外键)
└──────┬──────┘
       │ N:1
       ↓
┌─────────────┐      ┌─────────────┐
│ 需求主题    │ 1:1  │ 优先级评分  │
│  - ID       │────→ │  - 影响范围 │
│  - 标题     │      │  - 商业价值 │
│  - 聚类反馈 │      │  - 开发成本 │
│  - AI标签   │      │  - 总分     │
│  - 状态     │      └─────────────┘
└─────────────┘
```

### 关键设计决策
- **客户实体是必须的** - 没有客户数据，无法计算"商业价值"
- **反馈与需求主题是 N:1 关系** - 多条反馈聚合成一个主题
- **优先级评分独立存储** - 方便后续调整算法而不影响原始数据

## 四、MVP 功能清单 (Only What Matters)

### ✅ 必须实现的核心功能

#### 1. 数据导入模块

**功能 1.1: Excel/CSV 批量导入**
- 支持标准格式：`反馈内容 | 提交时间 | 客户名称 | 客户类型`
- 自动创建不存在的客户记录
- 导入后立即触发 AI 处理

**功能 1.2: 手动录入表单**
- 必填字段：反馈内容、客户名称
- 可选字段：客户类型、反馈来源
- 支持多行文本输入（最多 1000 字）

**技术要求：**
- 导入速度：500 条/分钟
- 支持格式：.xlsx, .csv (UTF-8 编码)
- 错误处理：逐行校验，跳过无效数据并生成错误报告

---

#### 2. AI 智能处理中枢 (核心壁垒)

**功能 2.1: 语义聚类 (Semantic Clustering)**
- 输入：所有"未聚类"状态的原始反馈
- 输出：需求主题列表，每个主题包含 N 条相似反馈
- 算法逻辑：
  1. 使用 LLM 提取每条反馈的 Embedding 向量
  2. 使用向量相似度计算（Cosine Similarity > 0.75）进行聚类
  3. 为每个聚类生成主题标题（15 字以内）

**功能 2.2: 自动分类与打标**
- 固定标签体系：`Bug` | `体验优化` | `新功能` | `性能问题` | `其他`
- AI 为每个需求主题打上 1-2 个标签
- 如果 AI 不确定（置信度 < 0.7），标记为"待人工确认"

**功能 2.3: 紧急程度识别**
- 二元分类：🔥 紧急 / 📝 常规
- 判断依据：反馈中是否包含强烈负面情绪词汇（崩溃、无法使用、严重影响等）
- 紧急反馈会在列表中标红显示

**功能 2.4: 内容摘要生成**
- 为每条长反馈（>50字）生成 20 字以内的摘要
- 摘要格式：`[客户名称] 反馈 [核心问题]`

**技术要求：**
- AI 模型选择：优先使用 DeepSeek/Qwen（成本考虑），失败则降级到 GPT-4o-mini
- 处理速度：100 条反馈 < 30 秒
- 异步处理：导入后台任务队列，前端轮询状态

**质量保障机制 (关键):**
- PM 可以查看每个聚类的详细反馈列表
- PM 可以手动拆分聚类（选择部分反馈创建新主题）
- PM 可以手动合并聚类（拖拽合并两个主题）
- 系统记录所有人工调整操作，用于后续模型优化

---

#### 3. 优先级评分引擎 (简化版)

**评分维度：**

| 维度 | 选项 | 分值 |
|------|------|------|
| **影响范围** | 个别用户 / 部分用户 / 大多数用户 / 全部用户 | 1 / 3 / 5 / 10 |
| **商业价值** | 普通客户 / 付费客户 / 大客户 / 战略客户 | 1 / 3 / 5 / 10 |
| **开发成本** | 1天 / 3天 / 1周 / 2周+ | 1 / 3 / 5 / 10 |

**计算公式：**
```
优先级总分 = (影响范围 × 商业价值) / 开发成本

特殊规则：标记为🔥紧急的需求，总分 × 1.5
```

**评分流程：**
1. 系统展示需求主题列表
2. PM 逐个为需求主题填写 3 个评分维度（下拉选择）
3. 系统自动计算总分并重新排序
4. PM 可随时修改评分，列表实时更新

**技术要求：**
- 评分数据独立存储（不直接修改需求主题表）
- 支持批量修改同一维度（如：将 5 个需求的"开发成本"都改为"1天"）

---

#### 4. 基础状态追踪

**状态流转：**
```
待处理 → 计划中 → 进行中 → 已完成
   ↓
 已忽略 (可恢复)
```

**功能要求：**
- 每个需求主题有且仅有一个状态
- 状态变更记录时间戳和操作人
- "已完成"状态显示完成日期
- "已忽略"需要填写忽略原因（可选）

**列表筛选：**
- 按状态筛选（多选）
- 按标签筛选（多选）
- 按优先级排序（降序）
- 按创建时间排序

## 六、技术架构约束 (Tech Constraints)

### 6.1 数据隔离（多租户设计）
- 采用 **Row-Level 隔离**（每行数据带 `tenant_id`）
- 所有查询必须强制带租户过滤条件
- MVP 阶段不考虑跨租户数据分析

### 6.2 AI 模型选择
- 优先级：DeepSeek V3 > Qwen-Turbo > GPT-4o-mini
- 成本控制：单次处理 < ¥0.5（100 条反馈）
- 必须支持异步处理（防止前端超时）

### 6.3 向量存储方案
- 方案 A：PostgreSQL + pgvector（成本低，适合 MVP）
- 决策依据：10 万条反馈的查询性能测试

### 6.4 性能指标
- 导入 100 条反馈：< 5 秒
- AI 聚类 100 条反馈：< 30 秒
- 列表加载（100 条需求主题）：< 1 秒
- 支持单租户 10 万条反馈（MVP 上限）

## 七、成功标准 (Definition of Success)

### MVP 验证指标

**核心指标 (Must Have):**
1. **聚类准确率** ≥ 80%（PM 人工评估，无需调整的聚类比例）
2. **时间节省** ≥ 10x（整理 100 条反馈从 2 小时降到 10 分钟）
3. **用户满意度** ≥ 4/5（5 分制问卷）

**次要指标 (Nice to Have):**
- 5 个种子用户完成完整流程测试
- 平均每个用户导入 ≥ 50 条反馈
- AI 自动打标准确率 ≥ 70%

**失败信号 (Kill Criteria):**
- 聚类准确率 < 60%（用户需要大量手动调整）
- PM 反馈"还不如我自己整理快"
- AI 处理时间 > 1 分钟（用户等不及）

## 八、用户验证计划 (Validation Plan)

### Phase 1: 内部测试（1 周）
- 使用我们自己的历史反馈数据（预计 200-300 条）
- 验证数据导入、AI 聚类、评分流程的完整性
- 修复明显的 Bug 和体验问题

### Phase 2: 种子用户测试（2 周）
- 招募 5 个目标用户：
  - 2 个正在用 Excel 管理反馈的 PM
  - 2 个用过 Canny 但觉得不够用的人
  - 1 个 SaaS 创业公司创始人
- 要求每人导入 ≥ 50 条真实反馈
- 每周 1 次访谈收集反馈

### Phase 3: 决策点（测试结束后）
- **如果核心指标达标** → 继续开发 Roadmap 功能
- **如果核心指标不达标** → 分析原因：
  - AI 模型问题？→ 调整模型或算法
  - 功能设计问题？→ 重新设计交互流程
  - 伪需求？→ Pivot 或终止项目

## 十、风险与应对 (Risk Mitigation)

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| AI 聚类质量差 | 🔴 高 | 🟡 中 | 设计人工调整机制 + 多模型对比测试 |
| AI 成本超预算 | 🟡 中 | 🟢 低 | 使用国产模型 + 设置每日调用上限 |
| 用户不愿导入真实数据 | 🟡 中 | 🟡 中 | 提供脱敏工具 + 签署保密协议 |
| 技术栈不熟悉导致延期 | 🟡 中 | 🟢 低 | 预留 20% 缓冲时间 |
| 种子用户流失 | 🟡 中 | 🟡 中 | 提供免费使用 + 定期跟进 |
