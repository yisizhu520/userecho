## 一、核心假设 (The One Thing to Prove)

**我们要验证的核心假设：**

> "AI 能够将散乱的用户反馈自动聚类并生成准确的需求主题，帮助 PM 快速从数据中提取洞察，其效率和质量超过人工整理 10 倍以上。"

**如果这个假设不成立，产品的所有其他功能都没有意义。**

### MVP 目标用户

- 中小企业的产品经理 (PM)
- 客户成功 (CS) 团队负责人
- 创业公司创始人

### 典型使用场景

**现状痛点：**
- PM 每周要从微信群、Excel、邮件中整理 50-200 条反馈
- 花费 4-6 小时手动分类、去重、提取关键需求
- 无法量化"哪些需求最该优先做"
- 不知道哪些客户提的需求最多（可能流失风险高）

**期望效果：**
- 10 分钟导入 100 条反馈（手动录入 / Excel / 微信截图）
- 系统自动聚类成 10-15 个需求主题
- 一键查看按优先级排序的需求列表
- AI 自动生成洞察："本周应重点关注登录闪退问题（影响 3 个大客户）"

## 三、核心数据结构 (Data Model First)

在写代码之前，必须先定义清晰的实体关系：

```
┌─────────────┐
│   客户      │ (Customer)
│  - ID       │
│  - 名称     │
│  - 付费等级 │ [普通/付费/大客户/战略]
└──────┬──────┘
       │ 1:N
       ↓
┌─────────────┐
│ 原始反馈    │ (Feedback)
│  - ID       │
│  - 内容     │
│  - 来源     │ [手动录入/Excel导入]
│  - 提交时间 │
│  - 客户ID   │ (外键)
└──────┬──────┘
       │ N:1
       ↓
┌─────────────┐      ┌─────────────┐
│ 需求主题    │ 1:1  │ 优先级评分  │
│  - ID       │────→ │  - 影响范围 │
│  - 标题     │      │  - 商业价值 │
│  - 聚类反馈 │      │  - 开发成本 │
│  - AI标签   │      │  - 总分     │
│  - 状态     │      └─────────────┘
└─────────────┘
```

### 关键设计决策
- **客户实体是必须的** - 没有客户数据，无法计算"商业价值"
- **反馈与需求主题是 N:1 关系** - 多条反馈聚合成一个主题
- **优先级评分独立存储** - 方便后续调整算法而不影响原始数据

### 环节 4️⃣：需求统计分析 ❌ **核心缺失**

#### ❌ 待实现功能（MVP 必须）

**4.1 优先级评分引擎**

这是 **MVP 最关键的缺失功能**。没有优先级评分，用户无法量化"哪个需求最该做"。

**评分维度：**

| 维度 | 选项 | 分值 |
|------|------|------|
| **影响范围** | 个别用户 / 部分用户 / 大多数用户 / 全部用户 | 1 / 3 / 5 / 10 |
| **商业价值** | 普通客户 / 付费客户 / 大客户 / 战略客户 | 1 / 3 / 5 / 10 |
| **开发成本** | 1天 / 3天 / 1周 / 2周+ | 1 / 3 / 5 / 10 |

**计算公式：**
```
优先级总分 = (影响范围 × 商业价值) / 开发成本

特殊规则：
- 标记为🔥紧急的需求，总分 × 1.5
- 影响大客户的需求，商业价值 +2 分
```

**需要实现：**
- [ ] Topic 表新增字段：`impact_scope` | `business_value` | `dev_cost` | `priority_score`
- [ ] API 接口：`PATCH /api/v1/topics/{id}/priority`
- [ ] 前端界面：在 Topic 详情页/列表页添加评分表单
- [ ] 实时计算并排序：评分变更后，列表自动重排

**预计工期：** 2 天

---

## 五、MVP 不包含的功能 (Out of Scope)

以下功能在 MVP 阶段**明确不做**，避免分散精力：

### ❌ 数据源扩展（V1.1 考虑）

- **应用商店监听（App Store / 安卓商店评论抓取）**
  - 原因：需要爬虫或付费 API，法律风险和成本不确定
  - 替代方案：用户手动复制粘贴应用商店评论到 Excel 导入
  
- **嵌入式 Widget（H5 反馈表单）**
  - 原因：需要设计无登录流程、反垃圾机制、公开链接管理
  - 替代方案：用户通过截图或手动录入收集的反馈

- **全网舆情监控（微信/抖音/小红书主动抓取）**
  - 原因：反爬机制强、法律风险高、技术成本极高
  - 替代方案：截图智能识别（用户主动上传）

### ❌ 高级功能（V1.2 考虑）

- **Webhook/API 接口**：暂无集成需求
- **自动生成回复话术**：需要大量 Prompt 工程和质量验证
- **数据可视化图表（象限图、趋势图）**：MVP 阶段用表格展示即可
- **与 Jira/飞书等外部系统集成**：集成成本高，优先级低
- **多人协作（评论、@提醒）**：MVP 阶段单人使用
- **邮件通知**：可用飞书/钉钉替代
- **细粒度权限管理**：MVP 阶段默认所有人可见可编辑

## 六、技术架构约束 (Tech Constraints)

### 6.1 数据隔离（多租户设计）
- 采用 **Row-Level 隔离**（每行数据带 `tenant_id`）
- 所有查询必须强制带租户过滤条件
- MVP 阶段不考虑跨租户数据分析

### 6.2 AI 模型选择
- 优先级：火山引擎（豆包）> DeepSeek V3 > Qwen-Turbo > GPT-4o-mini
- 成本控制：单次处理 < ¥0.5（100 条反馈）
- 必须支持异步处理（防止前端超时）
- 支持多模态（Vision）用于截图识别

### 6.3 向量存储方案
- 采用 PostgreSQL + pgvector（成本低，适合 MVP）
- 向量维度：4096（火山引擎）或 768（OpenAI）
- 索引类型：IVFFlat（适合 10 万条以内的数据）
- 查询性能要求：10 万条反馈 < 1 秒

### 6.4 性能指标
- 导入 100 条反馈：< 5 秒
- AI 聚类 100 条反馈：< 30 秒
- 截图识别（单张）：< 15 秒
- 列表加载（100 条需求主题）：< 1 秒
- 支持单租户 10 万条反馈（MVP 上限）

## 七、成功标准 (Definition of Success)

### MVP 验证指标

**核心指标 (Must Have):**

1. **聚类准确率** ≥ 80%
   - 测试方法：PM 人工评估 50 个聚类结果
   - 标准：无需手动调整的聚类比例 ≥ 80%

2. **时间节省** ≥ 10x
   - 测试方法：整理 100 条反馈的耗时对比
   - 标准：人工整理 2 小时 → 系统处理 10 分钟

3. **优先级排序准确性** ≥ 70%
   - 测试方法：PM 评估系统推荐的优先级排序
   - 标准：前 10 个需求中，有 7 个确实是高优先级

4. **用户完整流程通过率** ≥ 80%
   - 测试方法：5 个种子用户完成"导入→聚类→评分→洞察"全流程
   - 标准：至少 4 个用户完整走完流程且无卡顿

**次要指标 (Nice to Have):**
- 5 个种子用户完成完整流程测试
- 平均每个用户导入 ≥ 50 条反馈
- AI 自动打标准确率 ≥ 70%
- 截图识别准确率 ≥ 80%（微信/小红书/App Store）

**失败信号 (Kill Criteria):**
- 聚类准确率 < 60%（用户需要大量手动调整）
- PM 反馈"还不如我自己整理快"
- AI 处理时间 > 1 分钟（用户等不及）
- 优先级评分完全无法指导决策

## 十、待决策的关键问题

### 产品层面

- [ ] **客户数据从哪里来？**
  - 方案 A：用户在 Excel 中手动维护客户列表
  - 方案 B：提供客户管理模块（CRM Lite）
  - **建议：** 方案 A（MVP 阶段），方案 B（V1.2）

- [ ] **优先级评分是否可选？**
  - 方案 A：强制 PM 为每个 Topic 评分（确保质量）
  - 方案 B：评分可选，系统基于反馈数量自动排序
  - **建议：** 方案 B（降低使用门槛）

- [ ] **如果 AI 聚类完全错误，用户能否"全部取消聚类"重新来？**
  - 方案 A：提供"批量取消聚类"功能
  - 方案 B：允许手动调整，但不支持批量重置
  - **建议：** 方案 A（MVP 必须）

### 技术层面

- [ ] **向量存储用 pgvector 还是 Elasticsearch？**
  - **当前方案：** pgvector（成本低，已实现）
  - **何时切换到 ES：** 单租户超过 50 万条反馈

- [ ] **AI 模型 API 如果超时怎么办？**
  - **当前方案：** 重试 2 次，失败则降级到下一个 Provider
  - **何时优化：** 如果超时率 > 5%

- [ ] **多租户如何防止 SQL 注入和数据泄露？**
  - **当前方案：** 所有查询强制带 `tenant_id` 过滤条件
  - **技术实现：** SQLAlchemy ORM + Row-Level Security（PostgreSQL）
  - **审计：** 定期扫描慢查询日志

### 商业层面

- [ ] **MVP 是否收费？**
  - **方案 A：** 完全免费（种子用户测试阶段）
  - **方案 B：** 低价测试（¥99/月，前 10 个用户半价）
  - **建议：** 方案 A（MVP 验证阶段），验证成功后切换到方案 B

- [ ] **种子用户从哪里找？**
  - 渠道 1：朋友介绍（2-3 个）
  - 渠道 2：产品经理社群（ProductTank / 人人都是产品经理）
  - 渠道 3：冷启动用户（发帖招募）
  - **目标：** 5 个种子用户，2 周内完成招募

- [ ] **如果验证成功，下一步融资还是自力更生？**
  - **方案 A：** 自力更生（收费 + 慢速增长）
  - **方案 B：** 天使轮融资（快速扩张 + 招聘团队）
  - **决策点：** 种子用户测试结束后决定
